# Astrion OS — Lessons Learned

## Architecture
1. **Never use RGBA transparency without a compositor.** All `rgba()` in CSS and `gdk_screen_get_rgba_visual()` in C render as WHITE on X11 without compositing. Use solid colors only.
2. **`gtk_window_fullscreen()` needs a window manager.** Without a WM, use `gtk_window_resize(w, screen_width, screen_height)` + `gtk_window_move(w, 0, 0)` instead.
3. **`gtk_window_begin_move_drag()` needs a WM.** Without a WM, implement manual drag via motion-notify-event.
4. **GDK_SCALE=2 + GDK_DPI_SCALE compound.** Don't use both — GDK_SCALE alone handles HiDPI for native shell. Web content zoom is separate (webkit_web_view_set_zoom_level).
5. **ESM imports must be at the top of the file.** `import { X } from 'y'` inside a function or try/catch block crashes Node.js in ESM mode.
6. **C forward declarations required.** If function A calls function B and B is defined after A, add `static void B(...)` declaration at the top.
7. **C++ lambda syntax (`+[](...){}`) doesn't work in C.** Use named callback functions.
8. **A web page is NOT an operating system.** Users notice Ctrl+R, right-click menus, scroll bounce, and other browser behaviors. A native shell with WebKitGTK app windows is the right architecture.

## Live ISO
9. **Live ISOs lose ALL state on reboot.** localStorage, Wi-Fi profiles, settings — everything is in RAM overlay. Install to disk for persistence.
10. **System clock is wrong on live boot.** NTP needs internet, internet needs Wi-Fi, Wi-Fi needs manual setup. Use `ntpdate -u` (UDP, no SSL) in .xinitrc.
11. **DNS must be set explicitly.** Live ISOs often have empty `/etc/resolv.conf`. Write `nameserver 8.8.8.8` before anything network-dependent.
12. **SSL cert errors = wrong clock.** If `curl` fails with cert error (code 60), fix the date first: `sudo date -s "2026-04-07 12:00:00"`.

## Surface Pro 6
13. **Marvell Wi-Fi needs `firmware-misc-nonfree`.** The AVASTAR chip doesn't work without proprietary firmware blobs.
14. **Surface Pro 6 is 2736x1824 at 267 DPI.** Without scaling, everything is microscopic. GDK_SCALE=2 for native GTK, webkit_web_view_set_zoom_level(1.75) for web content.
15. **The `-nocursor` flag on `startx` permanently hides the cursor.** Don't use it. Use `xsetroot -cursor_name left_ptr` instead.
16. **`prompt()` may be blocked in WebKitGTK.** Use custom HTML dialogs instead of JavaScript `prompt()`, `confirm()`, `alert()`.

## Development
17. **`sudo nova-update` only updates web files (JS/CSS/HTML).** Compiled C binaries need a new ISO build or manual recompile on the device.
18. **Test on real hardware, not just in the browser.** Many features (battery, Wi-Fi, Bluetooth, audio) only work on the ISO.
19. **Don't hallucinate features.** Always verify with `grep` or `ls` that code actually exists before claiming it works.
20. **Cancel and retrigger builds rather than waiting for stale ones.** If you push a fix after triggering a build, the running build doesn't have the fix.

## Releases & ISO Publishing
21. **The `build-iso.yml` workflow only uploads to Actions artifacts, not to Releases.** Artifacts expire in 30 days and require login to download. For public distribution, the ISO must be manually uploaded to the Release via `gh release upload`. Fix pending: patch the workflow to auto-publish to release on tag push.
22. **Before publishing a stale ISO artifact, verify no `distro/` commits since the build.** Use `git log --oneline <artifact-sha>..HEAD -- distro/` — if it's empty, the ISO's shell/C code is still current. If non-empty, trigger a fresh build first.
23. **Always ship a SHA256 alongside any ISO.** Users need to verify integrity. `shasum -a 256 file.iso | tee file.iso.sha256` then upload both.
24. **GitHub release assets are hard-capped at 2 GB per file.** Check ISO size before attempting upload: `ls -lh file.iso`. Astrion ISO is 1.86 GB — safe, but close to the cap.
25. **Use `gh release upload --clobber` for new assets, not for replacing unrelated existing ones.** `--clobber` only overwrites assets with the same filename, leaves others alone.
26. **Name release assets with the product name + version + arch.** `astrion-os-0.1.95-amd64.iso`, not `nova-os.iso`. Makes the filename self-documenting and matches the current brand.

## Plan / Workflow
27. **The built-in `EnterPlanMode`/`ExitPlanMode` exists and is tighter than free-form plans in `tasks/todo.md`.** Plan mode enforces read-only exploration and a formal approval dialog before any edits. Use it for non-trivial tasks alongside `tasks/todo.md` (the todo file is the durable artifact, plan mode is the gate).
28. **Inversion thinking belongs in the plan itself, not just in the code.** For every task, list what will break, then invert each failure mode into the solution. This is more useful than a flat task list because it catches hidden assumptions before implementation.
29. **Commit large-file state to disk early, not at the end of a task.** Long sessions with many file edits can run out of context. Writing PLAN.md, presentation.md, etc. as soon as they're drafted (not buffered in chat) prevents losing work when context fills.

## Browser Preview & CSS Debugging
30. **`getComputedStyle(el).backgroundColor` can return stale mid-transition values.** When an element has `transition: background 0.3s` and you set its state via `dataset.brain = 's2'`, `getComputedStyle` read immediately after will return the interpolated (still in-progress) value, NOT the final target. This can make it look like CSS cascade isn't working when it actually is. Fix: either wait longer than the transition duration (e.g., 500 ms) before reading, OR use `el.computedStyleMap().get('background-color').toString()` AFTER the transition, OR use visual screenshot verification (most reliable). Inline style `!important` tests can be similarly confused by transition races.
31. **Leftover injected `<style>` tags and inline `element.style` from debugging pollute subsequent tests.** When running `preview_eval` multiple times, anything you inject persists across evals. Always remove injected styles and clear `element.removeAttribute('style')` before re-testing, OR reload the page between tests.
32. **The web Astrion OS boots through a login screen that only resolves on a click.** `js/boot.js` around line 278 has `loginScreen.addEventListener('click', ...)` that resolves the login promise when no password is set. In an automated preview environment (headless browser), the login screen never gets clicked, so `initMenubar()` and every downstream init never runs — the menubar DOM is there but has no handlers. When verifying behavior in preview: `document.getElementById('login-screen').click()` to advance past this gate. Affects every shell module that `boot.js` initializes after Phase 4.
33. **Astrion has a 5-minute idle screensaver.** If you step away from the preview mid-debug, the screensaver covers the desktop and you'll see stars + clock instead of your changes. Dismiss via `document.getElementById('screensaver').style.display = 'none'` or by triggering real input events (mousemove/keydown on the document).
34. **Files created by `sudo` in one CI step are root-owned and unusable by subsequent non-sudo steps.** `distro/build.sh` runs with `sudo`, which leaves `distro/output/nova-os.iso` owned by root. The next step in the workflow (`Publish ISO to latest GitHub release`) runs as the `runner` user and fails on `mv`/`rm` with `Permission denied`. Fix: add `sudo chown -R "$(whoami):$(whoami)" distro/output` at the top of any non-sudo step that touches files created by a sudo step. Cost is ~1 second and it's idempotent.
35. **`package.json` version drifts from git release tags in this repo.** The Electron auto-build pipeline (`build-desktop.yml`) bumps `package.json` independently, so it can say `1.0.0` while the latest release is `v0.1.95`. Always read version info from the latest release tag (`gh release list --limit 1 --json tagName`) instead of `package.json` when naming assets that must match the release. Strip the leading `v` with `VERSION="${LATEST_TAG#v}"`.
36. **"Fully opaque" comments can lie.** The radial glow in `on_desktop_draw` (nova-shell.c line ~569) had a comment "fully opaque, no compositor needed" but used `cairo_pattern_add_color_stop_rgba` with an alpha-1.0 center and alpha-0.0 edge. That IS an alpha gradient and still renders WHITE on X11 without a compositor (lesson #1). Don't trust comments — grep for `rgba`/`alpha` directly. Fix: use `cairo_pattern_add_color_stop_rgb` and pick the outer color to match the base gradient so there's no visible seam.
37. **Hardcoding `GDK_SCALE=2` in .xinitrc breaks VMs and normal displays.** On Surface Pro 6 (2736×1824) you need 2× scaling; in UTM/QEMU at 1920×1200 it makes everything 3.5× too big (combined with WebKit 1.75× zoom) and the setup wizard labels overlap horrifically. Fix: detect screen width from `xrandr`, apply `GDK_SCALE=2` only when width ≥ 2000px. User can override with `ASTRION_GDK_SCALE=1|2` env var.
38. **Fixed-column CSS grids (`repeat(3, 1fr)`) break at extreme zoom levels.** The setup wizard wallpaper picker used `grid-template-columns: repeat(3, 1fr)` inside a 560px max-width container. When the effective zoom was 3.5× in a VM, the grid cells overflowed and labels ("Geometry", "Forest", etc.) stacked on top of each other. Fix: `repeat(auto-fill, minmax(140px, 1fr))` so cells reflow, plus `min-width:0` on cells and `white-space:nowrap; overflow:hidden; text-overflow:ellipsis` on labels to truncate gracefully instead of overflowing.
39. **Read real-hardware values the same way across different subsystems.** `update_battery` reads `/sys/class/power_supply/BAT*/capacity` directly. `update_wifi` uses `popen("nmcli ...")`. `update_volume` uses `popen("pactl get-sink-volume ...")`. The pattern is identical: open, `fgets`, parse, `gtk_label_set_markup`. Reusing the pattern means no new libraries, no new dependencies, and any failure gracefully degrades to an "N/A" label. Good enough for M0.P1; libpulse integration can land in M0.P2 if we want real sliders.
40. **Verify multi-part Edit sequences after EACH step, not just at the end.** During M0.P1 Surgery 3, an Edit tool call used `new_str` instead of `new_string` (typo) and silently failed, so only HALF of the surgery landed. The call-site in the panel got added but the function definition did not, which the linker caught ~20 minutes later when CI built the ISO. Cost: one failed CI run (~27 min). Fix: after every Edit in a multi-step operation, grep for both the new symbol AND its expected use (`grep -n "update_volume" nova-shell.c` would've shown only the call sites with no definition).
41. **GtkLabel doesn't receive button-press events on its own — wrap in GtkEventBox.** M0.P2 Wi-Fi/Bluetooth/volume pickers all needed click handlers on menubar labels. Pattern: `GtkWidget *label = gtk_label_new(...); GtkWidget *evbox = gtk_event_box_new(); gtk_container_add(GTK_CONTAINER(evbox), label); gtk_widget_set_events(evbox, GDK_BUTTON_PRESS_MASK); g_signal_connect(evbox, "button-press-event", ...)`. Always pack the eventbox (not the label) into the parent container.
42. **`gtk_dialog_run()` is blocking modal — use it for simple yes/no flows like password prompts.** For the Wi-Fi password dialog, `gtk_dialog_run()` blocks until user clicks a button and returns the response code. Way simpler than connecting `response` signals. Only use for short interactions; never for long-running dialogs.
43. **Forward-declare new static functions at the top of nova-shell.c to keep the file navigable.** Put new `static` function prototypes in the "forward declarations" block near line 220. This lets you call them in any order and also documents all exported symbols in one place.
44. **Cancel superseded CI builds to save runner minutes.** When pushing multiple commits in a row, each push triggers a build but only the LATEST build matters. Cancel the in-flight older ones with `gh run cancel <run_id>` to save ~25 minutes per cancel. Rule of thumb: if you pushed a commit within the last 5 minutes of a running build, cancel the running one.
45. **Register ALL apps in native-mode boot, not a subset.** `js/boot.js`'s native branch was registering 16 of 52 apps. Result: launching any of the other 36 from nova-shell failed silently with "App not found". Always mirror the full app list in both the native and non-native boot paths, or extract them into a shared `registerAllApps()` helper.
46. **`pactl` output format varies slightly between versions.** The volume parser walks BACKWARD from the `%` character to find the integer, which tolerates different spacing. Never hard-code column positions — use anchor characters like `%` or `:` and walk relative to them.
47. **Run capability side effects BEFORE the notification, and wrap notifications in try/catch.** In M1.P2, several capability providers called `notifications.show()` at the end. When tested end-to-end in a context where the notification host DOM node didn't exist yet, the notification threw and the whole capability reported ok=false — even though the data side-effect had already succeeded. Fix: a `safeNotify()` helper that wraps the call in a try/catch. Principle: UI feedback is nice-to-have; never let it fail the underlying action.
48. **Always verify the exact storage shape the target app reads.** The notes.create capability initially wrote `{ id, title, body }` but Notes.js reads `note.content.split('\n')` — it expects a `content` field with the title as the first line. This only showed up in E2E testing. Fix: read the target app's source BEFORE writing to its storage. Lesson: "the shape you invent" almost never matches "the shape the consumer already uses."
49. **Structure capability providers so the data write happens FIRST, then the UI update.** If a notification fails or the app fails to launch, the data is still persisted and can be recovered on next open. Never couple "did the work happen" to "did the toast show up."
50. **Intent parser: handle verbs with optional targets explicitly.** "navigate to github.com", "explain recursion", "compute 5+3" all have no grammatical target — the parser needs a VERB_WITHOUT_TARGET set so these don't fail parsing. Learned during M1.P1 testing when `navigate to github.com` returned null because "github.com" isn't in TARGET_SYNONYMS.
51. **For free-form verbs (explain, summarize, ask), treat the entire remainder of the query as the topic.** Users write "explain recursion" expecting "recursion" to be the topic, not a target. In the arg extractor, set `args.topic = argsText` for these verbs before running the preposition patterns. This one line makes the parser feel natural instead of strict.
52. **Sprint compression is possible when the foundation is already solid.** M1 was scheduled for ~3 weeks per PLAN.md. It shipped in one day because M0 had already built the substrate (process-manager, file-system, event-bus, existing apps to use as backends). The lesson: invest heavily in the foundation — the rewards compound in every later milestone. M0 took ~10 days of intense work; M1 took ~4 hours because M0 had done the hard part.
53. **`crypto.subtle.digest` is async — NEVER `await` inside an open IndexedDB transaction.** IDB transactions auto-commit on the next macrotask. If you `const tx = db.transaction(...)` and then `await crypto.subtle.digest(...)` inside the callback, the microtask yield lets the txn commit (empty) before your hash resolves, and the next `tx.objectStore(...)` throws `TransactionInactiveError`. Fix: compute ALL async values (hashes, structuredClones of large blobs, network data) BEFORE opening the transaction. Inside the txn callback, use ONLY synchronous IDB calls. Learned while building graph-store.js for M2.P1.
54. **Emit events from `tx.oncomplete`, never from `request.onsuccess`.** An earlier request in a txn can succeed while a later request aborts the whole txn. If you emit `graph:node:created` from `onsuccess`, subscribers who call `getNode` inside the handler get back nothing (the abort rolled everything back). Only `oncomplete` fires after the txn is durable. Test this invariant directly: subscribe to the event, call `getNode` inside the handler, assert non-null. Caught during sanity-test design for graph-store.js.
55. **Fail-fast validation must live at NORMALIZE time, not EVALUATE time.** graph-query.js initially only caught unknown operator typos (`{ foo: { lessThan: 5 } }`) when `matchFilter` actually ran on a node. If the `from` type had zero nodes, the filter never ran and the typo was silently accepted. Fix: walk `where` at normalize time and reject any plain-object rhs with unknown keys, non-array `in`, non-boolean `exists`, etc. Rule of thumb: if an error can be detected from the QUERY alone without touching the data, detect it there. This also gives callers deterministic error behavior regardless of data state. Caught via an M2.P2 smoke test on an empty type.
56. **Prototype pollution defense requires BOTH normalize AND walker layers.** When walking a dotted field path like `'props.title'`, using `seg in cur` instead of `Object.prototype.hasOwnProperty.call(cur, seg)` means `'props.toString'` matches `Function.prototype.toString` and every node "has" it. ALSO: an attacker can pass `where: { '__proto__.polluted': 1 }` which needs to be rejected at query-parse time, not relied on at walk time. Both checks are cheap; both are necessary. Single-layer defenses leak. Caught via M2.P2 Plan agent review.
57. **One-shot migrations need a flag AND a graph-empty safety check.** graph-migration.js uses `localStorage.getItem('astrion-graph-migrated-v1')` as the primary flag, but also checks that the graph has zero nodes of the migrated types before running. Belt-and-suspenders: if someone dev-wipes the flag but the graph has data (from a previous run or manual seed), the safety check prevents double-migration. If someone dev-wipes the graph DB but leaves the flag, the flag still prevents re-running (which is usually what you want — manual resets are rare). This dual-gate pattern means "clean" resets need `window.__astrionMigrationReset()` to clear both.
58. **Graph-backed apps must subscribe to `graph:*` events for external-write consistency.** When Notes is rewired to read from the graph, and the Intent Kernel creates a note via `notes.create` capability, the capability writes to the graph but Notes's local cache is stale. Fix: subscribe to `graph:node:created` and refetch (cheap) when the type matches. Without this, a user who says "make a note called shopping" via Spotlight while Notes is open sees nothing change until they refresh. Don't couple the app to its own storage layer — couple it to the global mutation events.
59. **IDB transaction stability vs. JS engine timer drift → don't trust sub-millisecond createdAt ordering on rapid-fire creates.** Migration created 3 todos in a tight loop. Each `createNode` ran its own IDB transaction (~1-3ms). But the `Date.now()` values captured inside each createNode sometimes collided (all three landed in the same ms), which meant `orderBy: createdAt asc` returned them in arbitrary (IDB key-path) order instead of insertion order. For sub-millisecond ordering guarantees you need either an auto-increment sequence number or an explicit sleep between writes. For M2 I accepted the ordering jitter — the apps use `createdAt` ordering which is "close enough" for user-facing timelines.
60. **The rewind semantics trap: rewinding an update creates a NEW version, not a version rollback.** graph-store's `rewindMutation` on an `update_node` calls `updateNode(id, before.props)` which bumps the version counter. The node's `props` match the old state but `version` is higher than the original (e.g., v1 → v2 via user edit, v3 via rewind). This preserves the immutable provenance chain at the cost of the version counter being "append-only". Documented in the rewind method's doc comment. Callers who want EXACT version rollback need to wait for true snapshot restore in M5+.
61. **Day-by-day verification is cheaper than end-of-sprint verification.** M2 Days 3→6 were compressed into one morning. At every day boundary (migration, Notes wire, Todo wire, Reminders wire, capability-providers rewire, rewind), I ran `preview_eval` smoke tests BEFORE moving on. Day 4a caught a timing issue (700ms wasn't enough past the 500ms debounce); Day 4c caught that capability-providers.js still wrote to localStorage (bypassing the graph); Day 5 caught that rewinding deleteNode needed a low-level `_restoreNode` helper for same-id restoration. Each of those would have been a nightmare to debug if discovered at the end after 3 hours of unverified changes. The rule: every layer of new coupling gets its own verification before the next layer lands.
62. **Math detection must pre-empt verb lookup.** The intent parser's "what is 42 * 17" bug sat in the console for multiple sessions because the math-detection regex required the whole query to be a pure math expression, so "what" matched the `explain` verb synonym before math was considered. Fix: run a math pre-check BEFORE the verb scan. Strip common leading phrases ("what is", "what's", "calculate", "calc", "compute", "solve", "how much is", "tell me") + trailing `?`/`.`, then check if the remainder is a pure expression. This is a general rule: when one parser stage can shadow another, make the MORE SPECIFIC stage run first. Don't trust fallback ordering.
63. **Features you "think" are missing are often already built.** Day 1 exploration for the Polish Sprint found that the menubar clock with mini-calendar, battery percentage display, window minimize/maximize animations, focus-visible indicators, dock keyboard nav, notification history panel, Finder drag-drop, Finder inline rename, and Finder text/image previews were ALL already implemented. My original triage said "medium-large" for most of these. The ACTUAL Polish Sprint work was ~30% of the advertised scope — the rest was verification, polish, or renaming nova→astrion. The lesson: before planning a sprint, spend 20 minutes exploring the code for "already there" instead of 20 hours rebuilding. Exploration budget pays for itself 5x.
64. **An unfixed regex bug can survive 50+ code reviews.** The `"what is 42 * 17"` failure in `intent-parser` showed up in console logs for AT LEAST 4 previous commits across multiple sessions (check the git blame on the test case). Nobody — including me — fixed it until it became Day 1 of the Polish Sprint. When test failure output becomes "background noise" (1/19 always red), you stop seeing it. Fix: the regression-check at the top of every new commit should verify EVERY test suite shows "all N pass", not just "the one I touched." If a test has been red for multiple commits, FIX IT or DELETE IT — don't let it live in the background.
65. **`update-electron-app` is a complete auto-update solution; don't rebuild it.** Day 6-7 started as "integrate electron-updater from scratch." Within 5 minutes of exploration, I found `update-electron-app` was already in `installer/package.json` as a dependency AND wired in `installer/main.js:setupAutoUpdater()`. The "missing" work turned out to be: forward Squirrel events to the renderer via IPC, add a manual check-for-updates path, wire notification center, write the docs, and rebrand. None of that required touching the updater itself. Lesson: Electron's ecosystem has canned solutions for auto-update, code-signing, packaging, crash reporting, and deep-linking. Always grep `node_modules/` and `package.json` before writing anything from scratch.
66. **Accessibility infrastructure is cheap; accessibility verification is expensive.** Day 8 added 13+ aria-labels, role attributes, tabindex, a focus-trap utility, and retrofitted one dialog. All of that took ~30 minutes. But REAL accessibility — testing with VoiceOver on macOS, Orca on Linux, NVDA on Windows, real keyboard-only navigation of every screen — is a multi-session effort that requires being able to HEAR the screen reader output. For v0.2 the right move is: ship the infrastructure, don't claim "screen reader compatible" until it's been tested. False accessibility claims are worse than no claims — they mislead users who genuinely depend on assistive tech.
67. **Focus traps need capture-phase listeners.** A naive focus trap uses `document.addEventListener('keydown', handler)` (bubble phase). Dialog children can call `stopPropagation()` in their own keydown handlers (e.g., an input canceling Enter/Escape) and the trap never runs. Fix: use `document.addEventListener('keydown', handler, true)` — capture phase runs BEFORE any descendant listener. The tradeoff is that you have to be careful about `e.preventDefault` so you don't break inputs.
68. **`requestAnimationFrame` before calling `.focus()` on a newly-mounted element.** If you call `el.focus()` synchronously after `document.body.appendChild(dialog)`, the browser hasn't painted the dialog yet; scroll-into-view can misbehave and screen readers may not pick up the focus change. Wrapping `focus()` in a `requestAnimationFrame` callback lets the browser paint once before focusing. Tiny detail, big UX difference on slower devices.
69. **Window Management API requires explicit user permission — never pre-fetch on boot.** `window.getScreenDetails()` (Chromium 100+, needed for multi-monitor positioning) triggers a permission prompt the first time it's called in a browsing session. If you call it in `windowManager.init()`, every user sees a "Allow this site to manage windows on all your displays" prompt on first load — a terrible first impression. Correct pattern: expose a lazy `refreshScreenDetails()` method and only call it when an app explicitly needs multi-display info (Mission Control, Display Settings, etc.). The fallback single-display path uses `window.innerWidth/Height`, which is correct for single-monitor setups anyway.
70. **Sprint compression works when a prior sprint already did the foundation work.** Lesson #52 applied to M1. Lesson #63 is its flip side. The Polish Sprint that was budgeted for 9 days shipped in 1 day because (a) M0/M1/M2 had already built so much of the underlying infrastructure, and (b) Viraaj's checklist conflated "this exists" with "this needs to be built." Every sprint should start with an exploration pass that catalogs what's already there. If the actual delta is less than 30% of the advertised scope, budget accordingly — don't pad the timeline just because the list looks long.

## Agent Core Sprint (v0.3)
71. **ESM module identity is keyed by the FULL RESOLVED URL, not by file path.** When `preview_eval` called `import('/js/kernel/event-bus.js')` (absolute) and boot.js had `import './kernel/event-bus.js'` (relative), both should have resolved to the same URL and therefore been the same module instance. But adding a query string like `'/js/kernel/intent-executor.js?v=2'` for cache busting DOES create a new URL → a new module graph → a completely separate instance (different eventBus, different aiService singleton, different capability registry). This caused hours of phantom "why isn't my listener firing" debugging during Agent Core Sprint Phase 6. Rule: (a) never query-string-bust imports if you're depending on singleton identity with already-loaded code, (b) always reload the page to pick up a changed module, and (c) when debugging event-bus-crossing code, first confirm both sides are on the same module instance with `a.eventBus === b.eventBus`.
72. **`localStorage['nova-ai-provider']` can silently neuter every AI path.** During Agent Core Phase 6 verification, `aiService.ask()` was falling straight to `_mockResponse` without ever calling fetch to `/api/ai` or `/api/ai/ollama`. I spent ~45 minutes stubbing fetch, intercepting at the network layer, and re-reading the ai-service code before noticing the cause: `localStorage.getItem('nova-ai-provider') === 'mock'`, probably set by a Settings > AI test in a prior session. With `provider = 'mock'`, both `if (provider === 'auto' || provider === 'ollama')` and `if (provider === 'auto' || provider === 'anthropic')` skip, and the code falls straight to the offline mock string. Fix for verification: `localStorage.removeItem('nova-ai-provider')` before any AI-dependent test. Structural fix for later: raise a visible warning in the menubar brain indicator when provider is forced to 'mock' so users know why nothing works.
73. **Stub the singleton method, not the global.** To simulate Claude's response in a preview environment without an API key, I tried three approaches: (a) `window.fetch = stub`, (b) importing aiService and replacing `aiService.ask`, (c) replacing `aiService._mockResponse`. Only (c) worked cross-module because: (a) the stub misses when `provider='mock'` (see #72); (b) the import returned the same singleton BUT only because the URL matched exactly (lesson #71); (c) patches the singleton's own prototype method so every module sees the same override. Lesson: when patching for tests, prefer patching the concrete method on the singleton instance over swapping globals or intermediate layers. It's resilient to module-graph surprises and provider-config accidents.
74. **The intent-planner's two biggest risks are the same risk: structural validation of LLM output.** Malformed JSON and hallucinated capability ids are both "the LLM said something that isn't true about the system." The planner handles both in the same place: `tryParseJSON()` tolerates markdown fences + leading prose, then `validatePlan()` walks every step and calls `getCapability(step.cap)` to ensure it's in the registry. One retry with the error echoed back covers the "LLM was sloppy" case; the second failure returns a graceful `{ status:'failed' }` that Spotlight surfaces as a soft error. Rule: every LLM boundary in Astrion needs a strict schema check and a bounded retry — no "best effort" parsing, no fall-through-and-pray.
75. **Plan preview gating on LEVEL.REAL is the M2 precursor to the full Socratic loop (M6).** The executor's `executePlan()` sums total irreversibility tokens up-front and, if any step is L2+, emits `plan:preview` and waits for `plan:confirmed` OR `plan:aborted` before running a single step. Spotlight renders the preview with yellow border + "⚠ Press ↵ Enter to confirm" header. This is cheap to implement (~40 lines in the executor + ~15 lines of Spotlight render code), costs nothing for pure-sandbox plans (folder/file creates are L1 and skip the gate), and gives the user a real choke-point on dangerous operations well before M6 ships the full red-team agent. Never defer safety primitives to "the right milestone" when the minimum viable version is this cheap.
76. **Event-driven step streaming is more ergonomic than async iterators for UI code.** Early draft: `for await (const step of executePlan(plan))`. That's clean in the executor but forces Spotlight into an async loop that's hard to abort cleanly and doesn't compose with the existing event bus pattern. Final design: `executePlan` emits `plan:step:start`, `plan:step:done`, `plan:step:fail`, `plan:completed`, `plan:failed` on the shared event bus. Spotlight subscribes in `initSpotlight` and re-renders the panel on every event. Benefits: (a) any code can subscribe — not just Spotlight — so the native-mode shell could hook the same events; (b) Escape-to-abort fits naturally as another emit; (c) no generator state to leak. The pattern reuses the M1 intent lifecycle events exactly. Consistency matters.
77. **Heuristic routers beat "always use the LLM" for single-shot queries.** The planner is expensive (~1-3s Claude round-trip) and the existing parser is ~2ms. Routing every query through the planner would make "open terminal" feel broken. The `routeQuery` heuristic takes a query + parser result and picks fast-path OR plan-path in <1ms: compound markers (`and`, `then`, `;`), plus a verb-hints list for " and X" clauses, plus a confidence-threshold fallback for queries the parser rejected. This preserves the M1 snappy feel for the 80% single-capability queries while unlocking multi-step for the 20% that need it. Measure the cost of each path before routing — don't use the expensive path for everything just because it's "smarter." #52 / #70 applied to latency.
78. **Session-only conversation memory is the right privacy default.** Memory is stored in the M2 hypergraph as `conversation-turn` nodes, each tagged with a `sessionId` that rolls on 10-minute idle. The planner only ever fetches turns from the CURRENT session. Rejected alternatives: (a) persistent global memory leaks context across unrelated tasks ("I was helping yesterday with budgets" bleeds into today's coding); (b) no memory at all breaks "and now do the same for Documents" follow-ups. Session-only is predictable, graceful, and still M5-compatible (rewind works on turn nodes because they're just graph mutations). Don't let memory scope creep without a deliberate design decision — memories that outlive their purpose become privacy footguns.
79. **Binding substitution belongs in the EXECUTOR, not the planner.** I briefly considered having the planner emit fully-resolved args by inlining the expected folder path ("/Desktop/Projects") into step 2's `parent`. Rejected because: (a) the planner's expected path might not match the capability's actual output (e.g. if path resolution canonicalizes differently), (b) hard-coded paths make plans non-portable across reruns. Instead, step 1 declares `binds: 'folderPath'`, step 2 references `${binds.folderPath}`, and the executor resolves the binding from step 1's ACTUAL output right before running step 2. Generalization: anywhere a plan has cross-step data flow, make the upstream step name its output and the downstream step reference that name — don't let either side assume what the other produced.
80. **Every LLM boundary needs an offline fallback path that's verifiable.** The deliverable test couldn't rely on a real Claude API call because the preview sandbox has no `ANTHROPIC_API_KEY`. Instead of shipping code that "probably works when deployed", I verified the full planner → executor → files on disk → Spotlight panel pipeline by stubbing `aiService._mockResponse` to return a canned plan. This is NOT a substitute for real Claude testing — it's a complement. The real Claude round-trip has to be verified in a later session with a key set. Rule: build testability into the LLM boundary from day one (stubbable, inspectable, deterministic under stub) so you can ship the surrounding logic without waiting on credentials.

## Agent Core Sprint — follow-up review (v0.3-review)
81. **"Defensive readers, never throws" is a CLAIM you have to adversarially test, not a design assertion.** The prior session's retrospective said `summarizeContext()` in `context-bundle.js` "returns null, never throws." A follow-up review found a one-line path where it absolutely DID throw: `new Date(bundle.timestamp).toISOString()` blows up with `RangeError: Invalid time value` when `bundle.timestamp` is `undefined`/`null`/`NaN`. The error propagated up through `buildPlannerPrompt` → `planIntent` → the `intent:plan` handler's `catch` block, which only did a `console.warn` — so the whole plan silently died with zero `plan:*` events. The prior session never hit it because every call site in the sprint passed `getContextBundle()` (which always sets a real timestamp). Any adversarial test with a bare `{}` context would have caught it in 30 seconds. Rule: for every "this never throws" / "this never returns null" claim in a retrospective, write a test that deliberately feeds it the bad input BEFORE you write the sentence. The claim IS the test spec. The catch-and-console.warn pattern in an async event handler is especially dangerous because it silently hides these crashes from everywhere except the console.
82. **Unresolved template bindings should fail fast with a clean error, not leak into downstream capability args.** `resolveBindings()` in `intent-executor.js` has a deliberate "leave unresolved for validation" fallback where `${binds.nope}` stays literal if the upstream step didn't produce a matching binding. The intent was "the capability's validate will catch the garbage path." In practice it caught it — but with the confusing error `"Path outside allowed roots: /${binds.nope}/file.txt"`, which has nothing to do with why the plan really failed (a typo'd binding name in the planner's JSON output). Fix: scan the resolved args for any surviving `${binds.X}` tokens in `executePlan` BEFORE calling the capability, and emit `plan:step:fail` with `"unresolved binding: ${binds.X}"`. Generalization: anywhere you have a substitution layer that can leave sentinels in the output, add a post-substitution guard that fails with an error naming the sentinel. Downstream code has NO context about substitution — the failure message should never come from there.
83. **Stub verification + sanity tests ≠ shipped, even when the stub suite is thorough.** (see below for full text)
84. **Local LLMs ramble past the JSON close-brace if maxTokens is too generous.** The planner's `maxTokens: 1500` was tuned for Claude Haiku which stops cleanly after the JSON object. qwen2.5:7b kept generating prose explanations after the closing `}`, pushing response time past 30 seconds and sometimes triggering the AbortSignal timeout. Fix: cap at 500 tokens (a 5-step plan is ~300 tokens). Generalization: when you tune a parameter for one model and then swap models, re-evaluate every parameter that affects output length, temperature, and format compliance.
85. **Schema-valid but semantically wrong plans are harder to catch than schema-invalid ones.** "delete everything" produced a structurally valid plan that created a folder called `_delete_me` — the model obeyed "use only catalog capabilities" but MISAPPLIED them to approximate an intent it couldn't satisfy. The fix was adding a prompt rule: "if the user's intent cannot be satisfied by any combination of catalog capabilities, return clarify instead of forcing a plan." But this is a band-aid — the real defense is an intent-alignment validator that checks whether the plan's SEMANTIC effect matches the user's stated intent, not just whether the capability IDs are in the registry. That's M6 (Socratic loop) territory.
86. **Spotlight's `input.disabled` state must be reset on EVERY exit path, not just the happy path.** `handleSubmit` disables the input when entering the plan branch. `plan:completed` re-enables it after 1.2 seconds. But clarify, abort, and failure paths never re-enabled it — the input stayed disabled on the NEXT Spotlight open, locking the user out permanently. Fix: `open()` and `close()` both set `input.disabled = false`; the `plan:clarify` handler does the same before rendering choices. Rule: for any state flag that blocks user interaction (disabled, loading, locked), list every function that exits the interaction mode and assert the flag is reset in each one.
87. **Soak-testing with a weaker model is MORE informative than soak-testing with the model you tuned for.** The planner prompt was tuned for Claude Haiku 4.5. Testing with qwen2.5:7b found bugs that Haiku would have masked: maxTokens rambling (#84), semantically wrong plans (#85), and timing-dependent UI bugs (#86) that only surface when responses take 5-15 seconds instead of 1-3. A model that follows instructions 95% of the time validates the happy path. A model that follows instructions 70% of the time validates the ERROR paths — which is where real users live. Use the weaker model for soak testing, the stronger model for production.
