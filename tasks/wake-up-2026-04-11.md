## 🏗️ What got built

### `js/kernel/graph-store.js` (new, 849 lines)

IndexedDB-backed hypergraph storage. Singleton export: `graphStore`. Schema version 1 covers every index we'll need through Day 5 — no DB bumps required for M2 queries or rewind.

**API shipped:**
- **Mutations:** `createNode`, `updateNode`, `deleteNode`, `addEdge`, `removeEdge`
- **Reads:** `getNode`, `getNodeByContentHash`, `getNodesByType`, `getEdge`, `getEdgesFrom`, `getEdgesTo`, `getMutation`, `getMutationsSince`
- `init()` is idempotent (shared promise guard)

**Non-obvious things I made sure to get right:**
- **Hash BEFORE opening the transaction.** `crypto.subtle.digest` is async, and awaiting inside a txn callback auto-commits the txn before the hash lands. Every record is built in plain JS first, then one synchronous block writes everything.
- **Events fire from `tx.oncomplete`, never `request.onsuccess`.** Earlier tests I wrote (#11) specifically prove that by the time a `graph:node:created` handler runs, the node is already readable via `getNode`. If we'd fired events on `onsuccess`, subscribers could have seen stale data on a mid-txn abort.
- **Cascade event ordering.** `deleteNode` emits every `graph:edge:removed` event BEFORE the single `graph:node:deleted` event, so subscribers always see a consistent "node still here" → "edge gone" → ... → "edge gone" → "node gone" sequence.
- **Canonical hash form.** `sha256("node:v1\u001f" + type + "\u001f" + canonicalize(props))` where `canonicalize()` recursively sorts object keys. Never trust raw `JSON.stringify` key ordering — the spec only pins insertion order in modern V8/JSC but not everywhere.
- **No dedup on `createNode`.** The design doc's "content addressing gives free dedup" line was tempting but wrong for user documents. Two notes with identical content are two distinct nodes. We compute + store + index the hash, but `createNode` always inserts. Callers who want dedup can query `getNodeByContentHash` themselves. Comment in the file warns future-you not to "optimize" this.
- **Meta whitelist.** `createdBy` only accepts `{kind, brain, intentId, capabilityId}`. Prevents a sloppy or malicious capability provider from injecting arbitrary fields.
- **`structuredClone` on every return.** `getNode` returns a fresh clone; `before`/`after` snapshots in mutation records are clones. Callers can't corrupt the cache or history by mutating what they got back.
- **UUID fallback.** If `crypto.randomUUID` is missing (older WKWebView), uses `crypto.getRandomValues` to build a valid v4 UUID.
- **Every mutation method asserts `this.db`.** Cheaper than debugging `Cannot read 'transaction' of null` 10 minutes from now.

## 🧠 What didn't bite (and why that's a warning)

My inversion table had 20+ failure modes listed. Here's what I braced for that never happened:

- `TransactionInactiveError` from async hashing inside a txn — handled by pre-computing everything
- `onupgradeneeded` blocked by a stale connection from a previous session — nothing blocked
- Test DB leaking across hot-reloads — clean teardown every time
- `structuredClone` choking on `undefined` or `Date` in props — none of the test cases had those
- Cascade delete missing an edge — 6 edges cascaded cleanly across 3 runs
- LRU eviction firing at exactly the wrong moment — tests used fewer than 1000 entries

**The fact that nothing bit me on Day 1 is a warning sign**, not a victory. It probably means:
1. The sanity tests aren't stress-testing enough edge cases. Day 2 should add some.
2. Real usage (Day 4: wiring Notes/Todo to the graph with hundreds of existing entries) will hit things I haven't thought of yet.
3. I should specifically fuzz-test with: `Date` values in props, `undefined` fields, deeply-nested objects, circular references (should throw gracefully), and quota exhaustion.

Added nothing to `tasks/lessons.md` because nothing earned it. Day 2 will probably earn its first lesson.
